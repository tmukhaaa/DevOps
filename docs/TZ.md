# Техническое задание (п. 1)

**Тема:** Разработка контейнера для обеспечения безопасности приложения при угрозе «форсированного веб-браузинга» (159).

**Вариант приложения:** 2 — веб-приложение на Flask, развёртывание на облачной SaaS-модели (контейнеры Docker / оркестрация Kubernetes).

**Вариант алгоритма:** 3 — интеллектуальная обработка данных методом кластеризации K-Means с табличным и визуальным представлением результатов.

---

## 1. Основание и цель

Разработать защищённое многоконтейнерное веб-приложение, демонстрирующее:

- противодействие атаке forced browsing (подбор скрытых URL);
- обработку набора данных алгоритмом K-Means;
- хранение результатов в СУБД PostgreSQL;
- развёртывание в Docker Compose и Kubernetes (один Pod — четыре контейнера).

## 2. Назначение системы

Система предназначена для:

1. Приёма HTTP-запросов через reverse-proxy (nginx).
2. Выполнения кластерного анализа двумерных данных (x, y).
3. Отображения таблицы и scatter-графика кластеров.
4. Блокировки несанкционированных путей (/admin, /internal, path traversal).
5. Администрирования БД через pgAdmin (отдельный порт, не через публичный nginx).

## 3. Угроза и меры защиты

| Угроза | Описание | Мера |
|--------|----------|------|
| Forced browsing | Перебор URL (/admin, /backup, /.env) | Whitelist маршрутов во Flask; deny в nginx |
| Path traversal | Запросы с `../` | Блокировка regex в nginx и Flask |
| Information disclosure | Утечка версий, заголовков | `server_tokens off`, security headers |
| Прямой доступ к БД | Обход приложения | PostgreSQL только во внутренней сети Docker |

## 4. Состав системы (4 контейнера)

| № | Контейнер | Образ | Назначение |
|---|-----------|-------|------------|
| 1 | nginx | nginx:1.25-alpine | Точка входа, фильтрация URL, proxy |
| 2 | backend | backend (Flask) | K-Means, API, защита маршрутов |
| 3 | db | postgres:15-alpine | Хранение результатов анализа |
| 4 | pgadmin | dpage/pgadmin4:8 | Веб-интерфейс администрирования БД |

**Доступ к nginx:** порт `8080` (Docker), NodePort `30080` (Kubernetes).

## 5. Функциональные требования

- `GET /` — главная страница.
- `GET /analyze` — запуск K-Means (k=3), сохранение в БД, HTML-таблица + PNG.
- `GET /health` — JSON-статус приложения и последнего запуска.
- `GET /static/result.png` — визуализация после анализа.
- Запрещённые пути возвращают HTTP 403/404.

## 6. Требования к данным (п. 9)

- Формат: CSV, столбцы `x`, `y`.
- Объём: 1000 записей.
- Структура: 3 гауссовых кластера (центры согласованы с преподавателем).
- Генератор: `data/generate.py`.

## 7. Требования к развёртыванию

- **Docker:** `docker compose up --build -d`
- **Kubernetes:** `contig.yaml` — Pod с 4 контейнерами; ConfigMap `k8s/nginx-configmap.yaml`; Service в `contig.yaml`.

## 8. Нефункциональные требования

- Время отклика `/analyze` ≤ 10 с на 1000 точках.
- Логирование попыток forced browsing.
- Пароли БД — через переменные окружения (для учебного стенда допустимы demo-значения).

## 9. Критерии приёмки

1. Запускаются 4 контейнера, взаимодействуют по сети `secure-net`.
2. `/analyze` возвращает таблицу и график; данные в PostgreSQL.
3. `/admin`, `/internal` — 403.
4. Pod в Kubernetes в статусе Running.
5. Отчёт по пунктам 1–10 оформлен.
